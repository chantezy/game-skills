# 游戏QA测试工作流

## 概述

QA工作流的核心目标是**确保游戏的可玩性**。覆盖三个测试维度：剧情和数值测试、玩法和战斗测试、角色与单位测试。每个工作流聚焦于：如何发现可玩性问题、应该规避哪些常见问题、如何测试和解决。

---

## 一、可玩性保障框架

### 可玩性核心维度

| 维度 | 检查要点 | 常见问题 |
|------|---------|---------|
| 功能完整性 | 所有功能按设计正常运行 | 功能缺失、流程断裂、按钮无响应 |
| 数值平衡 | 难度曲线合理、经济系统平衡 | 过于简单/困难、通货膨胀、数值碾压 |
| 剧情合规 | 文本无错误、符合合规要求 | 错别字、敏感词、剧情矛盾、角色行为不一致 |
| 资源完整性 | 所有美术/音频资源就位 | 资源缺失、配置错误、资源与设定不符 |
| 体验流畅 | 玩家行为动线顺畅、无卡点 | 流程卡死、无法回退、信息不足 |
| 配置正确 | 配置表数据与设计一致 | 配置遗漏、数值错误、数据不一致 |

---

## 二、应规避的常见问题清单

### 数值类
- 难度断崖式跳跃（某关卡难度远超前后关卡）
- 经济系统崩溃（货币产出远大于消耗，或反之）
- 属性克制失效（某属性明显过强/过弱）
- 升级经验曲线不合理（后期升级过于困难/简单）

### 剧情类
- 角色行为与设定矛盾
- 剧情时间线错乱
- 世界观设定冲突
- 敏感内容/不合规表述
- 错别字与语法错误

### 配置类
- 角色/单位属性配置与设计文档不符
- 关卡配置参数错误（敌人数量、奖励数量等）
- 默认配置不合理（初始难度过高、引导不足）
- 数据不一致（前后端数据、策划案与配置表不匹配）

### 体验类
- 玩家卡在某个流程无法推进
- 关键信息提示不足导致玩家困惑
- 操作反馈不及时
- UI布局遮挡关键信息

## 三、测试工作流

### 工作流1：剧情和数值测试

#### 剧情测试

**测试目标：** 确保剧情文本合规、无错误，资源完整就位

**输入：** 剧情文案规范、敏感词汇要求、受众群体标准、错别字检查标准、资源就位检查要求

**AI处理步骤：**
1. **校验剧情文案**：逐段检查文本合规性、错别字、敏感词
2. **校验资源**：检查剧情涉及的场景资源、角色资源是否就位

**产出：**
- 文本合规性检查报告（列出所有问题文本及位置）
- 文案优化建议（修正方案）
- 资源配置状态报告（缺失资源清单）
- 资源完整性验证（通过/不通过）

#### 数值测试

**测试目标：** 确保数值配置正确、数据一致

**输入：** 数据一致性规范、数值表一致性检查要求、数据配置标准、数值验证规范

**AI处理步骤：**
1. **校验数值**：对比后端数据、数值表数据、策划案数据三者是否一致
2. **定位配置错误**：精确定位不一致的数据项

**产出：**
- 数值配置正确性报告（列出所有错误项）
- 数据一致性验证（通过/不通过，附差异清单）
- 配置错误定位（精确到具体配置项）

---

### 工作流2：玩法和战斗测试

#### 玩法功能测试

**测试目标：** 确保玩法功能按设计运行，用例覆盖完整

**输入：** 用例级别要求、验收时间标准、用例规范、线上BUG记录、测试需求说明

**AI处理步骤：**
1. **策划案解析**：拆解策划案中的功能点
2. **核心玩法总结**：提炼核心机制
3. **核心机制概括分析**：分析各机制的交互关系
4. **关键配置与设计意图提取**：识别需重点测试的配置项
5. **用例设计**：基于以上分析生成测试用例

**产出：**
- 测试用例脑图（功能点全覆盖）
- 案子设计建议（重点测试项标注）
- 功能覆盖分析（覆盖率百分比、未覆盖项）
- 风险点识别（高风险功能及原因）

#### 战斗测试

**测试目标：** 确保战斗难度合理、配置正确、平衡性达标

**输入：** 战斗难度标准、通用默认战斗配置、难度平衡要求、配置验证标准

**AI处理步骤：**
1. **战斗难度AI处理**：模拟不同玩家水平的战斗，评估难度曲线
2. **战斗配置AI处理**：检查战斗配置是否与设计一致

**产出：**
- 难度是否符合预期检查报告（附难度曲线图）
- 默认配置正确性验证（通过/不通过）
- 平衡性分析（属性克制是否有效、Boss难度梯度）
- 配置优化建议（具体调整方向和数值）

---

### 工作流3：角色与单位测试

**测试目标：** 确保角色/单位强度合理、配置正确

**输入：** 角色/单位强度标准、通用默认角色配置、强度平衡要求、配置验证标准

**AI处理步骤：**
1. **角色/单位强度AI处理**：模拟角色/单位在不同场景下的表现，评估强度档位
2. **角色/单位配置AI处理**：检查角色属性配置是否与设计文档一致

**产出：**
- 强度是否符合预期检查报告（附强度对比图）
- 默认配置正确性验证（通过/不通过）
- 平衡性分析（角色/单位间强度梯度是否合理）
- 配置优化建议（具体调整方向和数值）

## 四、测试用例设计规范

### 用例设计五原则

1. **边界优先**：优先测试边界条件（最小值、最大值、临界值）
2. **异常覆盖**：覆盖所有异常情况（网络断开、资源缺失、非法输入）
3. **流程完整**：覆盖玩家从开始到结束的完整行为路径
4. **交叉测试**：测试系统间的交互（如战斗中触发剧情、商店中购买后立即使用）
5. **回归测试**：线上BUG修复后需回归验证

### 测试优先级定义

| 级别 | 定义 | 执行时机 |
|------|------|---------|
| P0 | 阻断性问题（游戏崩溃、流程卡死） | 每次构建必测 |
| P1 | 核心功能问题（战斗无法进行、数值错误） | 每次版本测试 |
| P2 | 体验问题（UI错位、文案错误） | 版本测试 |
| P3 | 优化建议（体验改进、性能优化） | 视情况测试 |

---

## 五、问题解决流程

发现问题后按以下路径处理：

| 问题类型 | 解决方向 | 负责方 |
|---------|---------|-------|
| 数值错误/配置遗漏 | 修改配置表 → 验证 → 关闭 | 数值/系统策划 |
| 剧情矛盾 | 修订剧情文本 → 验证 → 关闭 | 剧情策划 |
| 资源缺失 | 补充美术资源 → 验证 → 关闭 | 美术 |
| 逻辑错误 | 修改代码 → 验证 → 关闭 | 程序 |
| 体验问题 | 调整设计/增加引导 → 验证 → 关闭 | 策划 |

配置可修复的问题直接修改配置验证后关闭；设计/代码问题提交对应负责人修改后回归验证。

## 与其他技能的协作

QA 测试案产出时,以下内容不在本技能范围内,由对应技能产出:

- **测试依据(各模块策划案的规则与数值)** → 由各策划模块(系统/数值/关卡/战斗/剧情/角色)产出
- **配置表校验规则(Schema 校验/边界值)** → 由 `game-tech-implementation`(规范层)产出
- **跨模块一致性测试用例** → 由 `game-full-workflow` 在组装时产出一致性报告,QA 基于此设计跨模块测试

**前置依赖(建议先完成):**
- 至少部分策划模块已产出(QA 测试基于具体规则设计测试用例)
- 配置表 Schema 已定义(配置校验基于 Schema)

**下游协作:**
- 测试发现的问题反馈到对应模块修正
- 测试报告作为 GDD 附录的"测试用例"章节
