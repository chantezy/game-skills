# 游戏全流程编排工作流

## 概述

本技能以游戏开发流程为主线，从**一句话或一段话创意点**出发，通过引导式讨论明确游戏性实现，整理成大纲供用户确认，再根据确认的大纲自动从模块池中选取所需策划模块，生成符合GDD（游戏设计文档）行业标准的完整游戏策划案。覆盖策划、美术、QA、程序全链路。

---

## 一、完整工作流总览

```
创意点（一句话/一段话/关键词）
    │
    ▼
AI解析创意点 → 提取核心要素
    │
    ▼
引导式讨论阶段（核心环节，讨论游戏性实现）
    ├── 讨论游戏类型与核心机制实现
    ├── 讨论目标受众与平台适配
    ├── 讨论视觉风格与情绪体验
    ├── 讨论核心循环与玩法深度
    └── 讨论风险与可行性
    │
    ▼
讨论充分性判断
    ├── 不充分 → 继续讨论缺失维度
    └── 充分 → 整理为游戏设计大纲
    │
    ▼
用户确认大纲
    ├── 不确认 → 标注修改点 → 回到讨论
    └── 确认 → 模块路由（自动选取所需模块）
    │
    ▼
逐模块产出策划案（每模块遵循三阶段）
    │
    ▼
按 GDD 模板组装为完整游戏策划案
```

**讨论原则：**
- 每次只讨论1-2个维度，不要一次性覆盖所有维度
- 不明确就给方向让用户选，提供2-4个可选方向
- 聚焦实现可行性，讨论"如何实现"而非"是否要做"
- 用户提供优先：参考内容由用户提供，未提供则AI基于游戏创意生成合适方向

## 二、模块架构

### 核心模块（几乎所有游戏都需要）

| 模块 | 职责 |
|------|------|
| 系统策划 | 玩法机制、操控方式、系统规则 |
| 数值策划 | 数值推导、经济平衡、成长曲线 |
| 美术生产 | 原画/场景/特效/动画/UI |
| QA测试 | 测试用例/配置校验/可玩性保障 |
| 程序开发 | 架构设计/代码生成/代码审查 |

### 类型模块池（按游戏类型按需选取）

| 模块 | 职责 | 适用条件 |
|------|------|---------|
| 关卡策划 | 关卡/动线/解谜/难度梯度 | 游戏有关卡、阶段或场景 |
| 战斗策划 | 战斗机制/Boss/平衡 | 游戏有战斗内容 |
| 剧情策划 | 世界观/剧情大纲/文本校验 | 游戏有叙事故事 |
| 角色与单位设计 | 角色/敌人/NPC/兵种/载具 | 游戏有可控角色或敌方单位 |

**不是所有游戏都需要全部模块。** 确认游戏类型后，AI自动从类型模块池中选取所需模块，用户可调整。例如：解谜游戏只需核心模块+关卡策划，不需要战斗和剧情；RPG则需要全部模块。

## 四、模块编排顺序

选取模块后，按以下建议顺序执行（用户可调整）：

```
1. 系统策划（确定核心系统和玩法框架）
2. 角色与单位设计（如有——确定角色/单位内容）
3. 剧情策划（如有——确定世界观和故事）
4. 关卡策划（如有——基于系统和内容设计关卡）
5. 战斗策划（如有——设计战斗机制和Boss）
6. 数值策划（平衡各系统的数值）
7. 美术生产（拆分美术需求，产出美术资产）
8. QA测试（测试验证）
9. 程序开发（技术实现）
10. 组装GDD
```

| 2.3 系统规则-经济 | 系统策划案 + 数值策划案 | 核心 |

| 3.2 关卡设计 | 关卡策划案 | 类型 |

## 六、GDD 四部分结构

### 第一部分：项目总览（High-Concept）
让所有人在5分钟内了解项目全貌。必含：游戏名称、游戏类型、目标平台、目标受众、游戏特色（3-5个卖点）、核心循环、视觉风格。

### 第二部分：核心机制（Core Mechanics）
GDD的心脏，要求**绝对精确、无歧义**。必含：操控方式、角色机制（移动/战斗/成长）、系统规则（经济/数值/AI）、用户界面(UI)。

### 第三部分：游戏内容（Game Content）
描述玩家能实际体验到的内容。必含：世界观与故事、关卡设计、角色与单位、物品与装备、音频设计。

### 第四部分：附录与规划（Appendix & Schedule）
必含：美术需求拆分、规则文档与配置、测试用例、技术实现方案、术语表、版本记录、未来规划。

## 七、三阶段产出模式

所有策划类工作流按三阶段推进：

### 第一阶段：需求大纲
明确关键词边界，拆解核心要素，产出系统/关卡目标。对应GDD第一部分项目总览。
产出包含：设计目标、核心关键词、系统目标、参考方向。

### 第二阶段：细化需求与原型设计
生成创新思路，设计关键模块与玩家行为，产出原型方案。对应GDD第二部分核心机制。
产出包含：独特机制、核心乐趣、潜在风险、关键模块表、原型设计。

### 第三阶段：完整策划案
整合产出完整GDD，含美术需求拆分与规则文档。对应GDD全四部分。
各模块产出按"模块到GDD章节映射表"填入对应位置。

## 八、评审检查清单

产出后对照以下三类检查项逐项评审：

**完整性检查：** 项目总览、操控方式、角色机制、系统规则、UI设计、世界观与故事、关卡设计、角色与单位、物品与装备、美术需求、术语表、版本记录是否齐全。

**精确性检查：** 数值精确（无模糊词）、逻辑无歧义（程序员可直接编码）、边界条件明确、公式完整可计算。

**可视化检查：** 逻辑流程有流程图、界面布局有线框图、数据用表格列举、视觉风格有参考图。

未通过检查项按反馈循环修正：不满意 → 说明不满意的具体点 → 优化 → 重新生成 → 问题录入风险记录 → 反哺回输入。

## 九、快速开始模式

对于创意已经成熟、不需要完整引导式讨论的用户，可以使用快速产出模式。

### 适用条件

用户提供以下信息中的**至少 4 项**：
- 游戏类型（如"ARPG"、"Roguelike"、"模拟经营"）
- 核心机制描述（一句话说明玩家主要在做什么）
- 目标平台和受众
- 视觉风格方向
- 参考游戏

### 快速流程

```
用户提供成熟创意（含上述4+项信息）
    │
    ▼
AI直接解析 → 生成游戏设计大纲（跳过逐维度讨论）
    │
    ▼
用户确认/微调大纲
    │
    ▼
按模块路由直接产出策划案（每模块仍遵循三阶段）
    │
    ▼
组装完整GDD
```

### 与标准流程的区别

| 维度 | 标准流程 | 快速模式 |
|------|---------|---------|
| 创意讨论 | 逐维度引导式讨论 | AI直接解析，用户确认 |
| 大纲产出 | 讨论充分后整理 | AI一次性生成 |
| 模块产出 | 三阶段（不变） | 三阶段（不变） |
| 适用场景 | 创意初期、需要探索方向 | 创意成熟、方向明确 |

## 使用方式

1. **从创意点开始讨论（推荐）**：提供创意点，通过引导式讨论明确方向
2. **快速产出模式**：创意已成熟时，提供4+项关键信息直接进入模块产出
3. **直接进入模块**：已确认大纲时，按"工作流路由"选择对应工作流
4. **组装完整GDD**：各模块产出后，按"GDD章节与模块映射表"组装
