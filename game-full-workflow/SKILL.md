# GDD 框架文档构建器与一致性校验器

## 概述

本技能是**GDD 框架文档构建器 + 跨模块一致性校验器**。核心场景是:用户已经产出了游戏的各个模块内容(系统策划案、数值策划案、关卡策划案、战斗策划案、剧情策划案、角色设计案、美术需求、QA 用例等),输入后由本技能进行**模块索引构建 + 跨模块逻辑校验 + 冲突检测**,最终输出一份**GDD 框架文档**(含模块索引表、校验报告、术语表、冲突报告)。

**关键定位(与"内容拼接器"的区别):**

本技能**不复制各模块的具体内容**,而是产出一份"框架文档",让团队通过这份文档知道:
- 完整 GDD 需要哪些模块(模块索引表)
- 每个模块的内容在哪(引用跳转)
- 各模块之间是否自洽(校验报告)
- 跨模块术语是否统一(术语表)
- 有哪些冲突待解决(冲突报告)

**为什么不复制内容?**
- 各模块内容只在各模块技能内维护一份,避免重复
- 改一个模块不需要同步改 GDD,框架文档的索引不变
- 本技能的独有价值是**跨模块校验**(任何单模块都做不了),不是"内容拼接"

**定位说明:**
- 本技能**不负责创意讨论**,也不负责单个模块的策划案产出
- **不假设用户都是从头到尾设计游戏**——用户可能在任意阶段带着已完成的模块内容来组装
- 核心价值是**让整个策划案有统一性**:各模块之间前后逻辑自洽、数值能算平、配置表字段一致、状态机不冲突
- 与其他技能的关系:
  - 创意发散 → `game-creative-discussion`(产出初步大纲)
  - 单模块策划产出 → `game-system-design` / `game-numerical-design` / `game-combat-design` / `game-level-design` / `game-narrative-design` / `game-character-design` / `game-art-production` / `game-qa-testing`
  - 技术实现规范 → `game-tech-implementation`
  - **本技能 = 构建模块索引 + 跨模块校验**

## 一、典型使用场景

### 场景 A:全模块组装
用户已完成所有模块策划,希望构建 GDD 框架文档,让团队通过索引查阅各模块内容,并确认各模块之间无冲突。

### 场景 B:部分模块组装
用户只完成了部分模块(如只有系统+数值+战斗),希望先构建已有部分的索引,识别缺失模块,并对已有部分做校验。

### 场景 C:一致性体检
用户已有完整 GDD(各模块内容分散维护),希望做一次"跨模块校验"找出不一致点,不需要重建索引。

### 场景 D:增量组装
用户之前已构建过一版 GDD 框架文档,现在新增/修改了某个模块,希望重新校验受影响的部分,并更新索引。

## 二、整体工作流

```
用户输入各模块策划案内容(或各模块文档的位置/引用)
    │
    ▼
模块完整性识别 ──→ 识别已提供模块 vs 缺失模块
    │
    ├── 缺失核心模块 ──→ 提示用户补产(不强制阻塞,可先构建已有部分索引)
    │
    ▼
构建模块索引表 ──→ 按 GDD 标准章节,标注每章内容来自哪个模块
    │                 (不复制内容,只做引用跳转)
    ▼
跨模块逻辑校验(核心环节)
    ├── 数值自洽性校验
    ├── 角色一致性校验
    ├── 配置表字段一致性校验
    ├── 状态机一致性校验
    ├── 经济系统闭环校验
    └── 术语/命名统一性校验
    │
    ├── 发现冲突 ──→ 输出冲突报告 + 解决建议 ──→ 用户选择处理方式 ──→ 回到校验
    │
    ▼ 通过
构建术语表 ──→ 汇总各模块专有术语,统一命名
    │
    ▼
自洽性评审清单复核
    │
    ▼
输出 GDD 框架文档(模块索引表 + 校验报告 + 术语表 + 冲突报告)
```

## 三、模块完整性识别

### 核心模块(组装前应检查)

| 模块 | GDD 章节归属 | 缺失时处理 |
|------|------------|-----------|
| 系统策划 | 2.1/2.2/2.3/3.4 | 提示补产,否则核心机制章节索引空缺 |
| 数值策划 | 2.3.2/4.2 | 提示补产,否则数值体系无法校验 |
| 美术需求 | 4.1 | 提示补产,否则美术拆分索引缺失 |
| QA 测试 | 4.3 | 提示补产,否则测试用例索引缺失 |
| 技术实现 | 4.4 | 提示补产,否则技术方案索引缺失 |

### 类型模块(按游戏类型按需)

| 模块 | GDD 章节归属 | 缺失时处理 |
|------|------------|-----------|
| 关卡策划 | 3.2 | 游戏有关卡但缺失 → 提示补产 |
| 战斗策划 | 2.2.2/2.3.3 | 游戏有战斗但缺失 → 提示补产 |
| 剧情策划 | 3.1 | 游戏有叙事但缺失 → 提示补产 |
| 角色与单位设计 | 3.3 | 游戏有角色但缺失 → 提示补产 |

> **原则:** 缺失模块不阻塞组装,但在 GDD 框架文档对应章节标注"待补产",并在校验报告中列为风险项。

## 五、GDD 四部分结构(索引文档目标)

### 第一部分:项目总览(High-Concept)
让所有人在5分钟内了解项目全貌。**本框架文档只写摘要 + 引用跳转**,完整内容见 `game-creative-discussion`。
- 必含摘要:游戏名称、游戏类型、目标平台、目标受众、游戏特色(3-5个卖点)、核心循环、视觉风格
- 详细内容:见 `game-creative-discussion` 产出的"游戏创意框架"

### 第二部分:核心机制(Core Mechanics)
GDD的心脏,要求**绝对精确、无歧义**。**本框架文档只做章节索引**,具体内容在各模块技能内维护。
- 必含章节:操控方式、角色机制(移动/战斗/成长)、系统规则(经济/数值/AI)、用户界面(UI)
- 各章节内容:见模块索引表对应模块

### 第三部分:游戏内容(Game Content)
描述玩家能实际体验到的内容。**本框架文档只做章节索引**。
- 必含章节:世界观与故事、关卡设计、角色与单位、物品与装备、音频设计
- 各章节内容:见模块索引表对应模块

### 第四部分:附录与规划(Appendix & Schedule)
**本框架文档只做章节索引**,附录内容在各模块技能内维护。
- 必含章节:美术需求拆分、规则文档与配置、测试用例、技术实现方案、术语表、版本记录、未来规划
- 术语表、版本记录:由本技能产出(跨模块汇总)
- 其他章节:见模块索引表对应模块

## 六、跨模块逻辑校验(核心差异化能力)

**这是本技能区别于"简单索引"的关键价值。** 各模块分散维护后,跨模块的一致性校验成为本技能的独有能力。校验时必须对以下维度做交叉校验,发现冲突即输出冲突报告。

### 6.1 数值自洽性校验

**校验目标:** 系统策划的经济系统 ↔ 数值策划的日产/日消比 ↔ 战斗策划的伤害公式,三方数值是否能算平。

**校验项:**

| 校验点 | 校验方法 | 冲突示例 |
|--------|---------|---------|
| 战斗数值闭环 | 用数值策划的公式参数,反推战斗策划的"打几轮/持续多久"设计意图是否成立 | 数值策划设定普通怪HP=440,但战斗策划设计意图是"打3轮",算出来要打4轮 |
| 玩家生存数值 | 玩家HP vs 怪物伤害,是否能承受战斗策划设计的"被攻击5次" | 战斗策划要求承受5次,数值推导只能承受3次 |
| 极端 Build | 数值策划的极端 Build 验证表是否覆盖战斗策划的所有 Build 类型 | 战斗策划有"坦克型"Build,但数值验证表没有 |
| 经济产出-消耗 | 系统策划定义的货币产出途径 vs 数值策划的日产估算是否一致 | 系统策划说每日任务给500金币,数值策划日产估算按800算 |
| 物品定价锚定 | 系统策划的物品定价 vs 数值策划的"时间价格"是否吻合 | 系统策划定价回复药100金币,数值策划按时间价格应是150金币 |
| 成长曲线 vs 内容解锁 | 数值策划的升级时长 vs 关卡策划的关卡推进节奏是否匹配 | 数值策划20小时到50级,关卡策划设计了40小时的主线关卡 |

### 6.2 角色一致性校验

**校验目标:** 角色设计案 ↔ 战斗策划技能 ↔ 美术需求,是否对得上。

| 校验点 | 校验方法 | 冲突示例 |
|--------|---------|---------|
| 角色技能数量 | 角色设计案定义的技能数 vs 战斗策划的技能配置表字段数 | 角色设计案有5个技能,战斗策划只配了4个 |
| 角色定位 vs 战斗机制 | 角色设计案的定位(坦克/DPS/辅助)vs 战斗策划的数值定位是否吻合 | 角色定位是坦克,但战斗数值给的高攻击低防御 |
| 角色外观 vs 美术需求 | 角色设计案的外观描述 vs 美术需求拆分的角色清单 | 角色设计案有10个角色,美术需求只列了8个 |
| 角色属性 vs 数值体系 | 角色设计案的属性初始值 vs 数值策划的属性成长曲线起点 | 角色设计案ATK初始50,数值曲线从100起算 |

### 6.3 配置表字段一致性校验

**校验目标:** 各模块引用的配置表字段是否命名统一、取值兼容。

| 校验点 | 校验方法 | 冲突示例 |
|--------|---------|---------|
| 字段命名统一 | 同一字段在各模块的命名是否一致 | 战斗策划用 `skill_mult`,数值策划用 `skill_multiplier` |
| 字段类型兼容 | 同一字段的类型定义是否一致 | 战斗策划 `cooldown` 是 float,系统策划是 int |
| 取值范围兼容 | 各模块对同一字段的取值范围假设是否冲突 | 战斗策划技能倍率0.8-3.0,数值策划验证用到了5.0 |
| 枚举值一致 | 枚举类型(如元素类型、技能类型)在各模块的定义是否一致 | 战斗策划元素有"光/暗",数值策划克制矩阵只有"火/冰/雷/风/地" |
| 默认值一致 | 同一字段在各模块的默认值是否一致 | 移动速度战斗策划默认3.0,系统策划默认2.5 |

### 6.4 状态机一致性校验

**校验目标:** 角色状态机 ↔ 战斗状态机 ↔ Boss 状态机,状态枚举是否冲突、转换条件是否互斥。

| 校验点 | 校验方法 | 冲突示例 |
|--------|---------|---------|
| 状态枚举冲突 | 不同状态机的状态名是否重复或含义冲突 | 角色状态机有 `Attacking`,战斗状态机也有 `Attacking` 但含义不同 |
| 转换条件互斥 | 同一触发条件下,不同状态机的转换目标是否冲突 | 受击时角色状态机转 `Hitstun`,但战斗状态机要求继续 `Attacking` |
| 状态优先级 | 高优先级状态(死亡/受击)是否在所有状态机中都正确覆盖 | 角色状态机有 `Dead` 转换,但战斗状态机没有从 `Attacking` 到 `Dead` 的转换 |
| 状态触发时机 | 状态进入/退出的时机是否在各模块描述一致 | 战斗策划说"攻击后摇0.2s后可闪避",系统策划的状态机要求0.3s后才能转 `Dodging` |

### 6.5 经济系统闭环校验

**校验目标:** 产出 → 消耗 → 成长形成闭环,无断层。

| 校验点 | 校验方法 | 冲突示例 |
|--------|---------|---------|
| 产出途径完整 | 系统策划定义的所有产出途径,数值策划是否都估算了日产量 | 系统策划有"挂机产出",数值策划日产估算没算 |
| 消耗途径完整 | 系统策划定义的所有消耗坑,数值策划是否都估算了日消耗 | 系统策划有"装备修理",数值策划日消估算没算 |
| 资产曲线合理 | 数值策划的30/60/90天资产预测 vs 关卡/角色解锁节奏是否匹配 | 30天净资产只能买中级武器,但关卡策划30天解锁了需要稀有武器的关卡 |
| 货币种类一致 | 系统策划定义的货币种类 vs 数值策划使用的货币是否一致 | 系统策划有"钻石"货币,数值策划定价表没有钻石定价 |

### 6.6 术语/命名统一性校验

**校验目标:** 同一概念在不同模块是否用词统一。

| 校验点 | 校验方法 | 冲突示例 |
|--------|---------|---------|
| 概念命名统一 | 同一游戏概念在各模块的称呼是否一致 | 战斗策划叫"体力",系统策划叫"耐力",数值策划叫"ST" |
| 数值单位统一 | 同一参数的单位标注是否一致 | 战斗策划速度单位"m/s",系统策划"米/秒",数值策划"格/帧" |
| 关卡/Boss 命名 | 关卡策划的关卡ID vs 剧情策划的章节命名是否对应 | 关卡策划 LV-001,剧情策划叫"第一章"无对应ID |
| 角色命名 | 角色设计案的角色名 vs 剧情策划的角色名是否一致 | 角色设计案"艾莉娅",剧情策划"艾莉亚" |

---

## 七、冲突检测与解决流程

### 冲突报告格式

发现冲突时,输出结构化报告:

```markdown
# 一致性冲突报告

## 冲突概览
- 总冲突数:X 个
- 严重等级分布:P0(阻断)X 个 / P1(重要)X 个 / P2(建议)X 个

## 冲突详情

### 冲突 1: [冲突标题]
- **严重等级:** P0/P1/P2
- **冲突类型:** 数值自洽 / 角色一致 / 字段一致 / 状态机 / 经济闭环 / 术语统一
- **涉及模块:** [模块A] ↔ [模块B]
- **冲突描述:**
  - 模块A 说:[具体内容]
  - 模块B 说:[具体内容]
- **影响:** [如果不解决会导致什么问题]
- **解决建议:**
  - 方案1:[建议 + 代价]
  - 方案2:[建议 + 代价]
- **用户决策:** [待用户选择]
```

### 严重等级定义

| 等级 | 含义 | 处理方式 |
|------|------|---------|
| P0 阻断 | 会导致程序无法编码或数值无法算平 | 必须解决才能完成组装 |
| P1 重要 | 会导致玩家体验明显异常 | 强烈建议解决,可在 GDD 中标注"待优化" |
| P2 建议 | 不影响功能但影响一致性 | 建议统一,可延迟处理 |

### 解决流程

```
发现冲突
    │
    ▼
输出冲突报告(含解决建议)
    │
    ▼
用户选择处理方式
    ├── 采纳方案1 → 应用修改 → 重新校验受影响部分
    ├── 采纳方案2 → 应用修改 → 重新校验受影响部分
    ├── 自定义修改 → 用户说明 → 应用修改 → 重新校验
    └── 暂不处理 → 标注为"已知问题" → 继续组装(GDD 中标注)
    │
    ▼
冲突解决记录(写入版本记录)
```

## 八、术语表(跨模块汇总产出)

**术语表是本技能的独有产出之一。** 各模块分散维护时,同一概念可能用不同称呼,本技能在组装时统一汇总并校验。

**术语表模板:**

```markdown
## 术语表

| 统一术语 | 别名(各模块原用词) | 含义 | 所属模块 |
|---------|------------------|------|---------|
| 体力 | 耐力 / ST / Stamina | 玩家执行动作消耗的资源 | 系统 / 数值 / 战斗 |
| 技能倍率 | skill_mult / skill_multiplier | 技能伤害的乘数系数 | 战斗 / 数值 |
| 普通怪 | 小怪 / Mob | 非Boss的普通敌人 | 战斗 / 关卡 / 数值 |
| ... | ... | ... | ... |
```

> **注:** 术语表在六维度校验的"术语/命名统一性校验"环节产出,冲突项需用户确认统一用词后再定稿。

## 九、自洽性评审清单

组装完成后,对照以下清单做最终复核:

### 完整性检查
- [ ] 模块索引表是否覆盖 GDD 四部分所有章节
- [ ] 核心模块章节是否均有索引(非"待补产")
- [ ] 类型模块章节是否按游戏类型索引或标注"不适用"
- [ ] 术语表是否收录所有模块的专有术语

### 精确性检查(引用模块内容的精确性)
- [ ] 索引指向的模块内容是否存在
- [ ] 索引标注的章节位置是否准确
- [ ] 各模块内容自身是否精确(无模糊词如"大概""可能")
- [ ] 边界条件明确(数值有 clamp,空值有兜底)

### 一致性检查(本技能核心)
- [ ] 数值自洽性校验全部通过(或冲突已解决/标注)
- [ ] 角色一致性校验全部通过(或冲突已解决/标注)
- [ ] 配置表字段一致性校验全部通过(或冲突已解决/标注)
- [ ] 状态机一致性校验全部通过(或冲突已解决/标注)
- [ ] 经济系统闭环校验全部通过(或冲突已解决/标注)
- [ ] 术语/命名统一性校验全部通过(或冲突已解决/标注)

### 可视化检查
- [ ] 模块索引表是否清晰可读
- [ ] 校验报告是否有冲突详情
- [ ] 术语表是否完整

未通过检查项按反馈循环修正:不通过 → 标注问题 → 返回对应模块修改 → 重新校验。

## 十、输出产物

组装完成后输出:

1. **GDD 框架文档** — 含模块索引表 + 项目总览摘要,不复制各模块具体内容,团队通过索引跳转查阅
2. **一致性校验报告** — 包含六维度校验详情、冲突检测结果、解决记录、剩余已知问题
3. **术语表** — 跨模块汇总统一所有专有术语命名(独有产出)
4. **冲突报告** — 结构化列出所有跨模块冲突及解决建议(独有产出)
5. **配置表清单** — 汇总各模块涉及的配置表列表(字段一致性已校验)
6. **状态机汇总** — 汇总所有状态机及其转换关系(冲突已解决)
7. **版本记录** — 记录本次组装的模块版本、冲突解决历史

> **关键:** 输出产物中,**GDD 框架文档是"索引"而非"内容容器"**。各模块的具体内容仍由各模块技能维护,本技能只做索引引用和跨模块校验。

## 十一、使用方式

### 输入要求
用户提供以下内容(至少部分):
- 已完成的各模块策划案(系统/数值/关卡/战斗/剧情/角色/美术/QA/技术),或各模块文档的位置/引用
- 可选:项目大纲(若没有,从各模块内容反推)
- 可选:已知冲突或关注点

### 调用示例
```
用户:我已经完成了系统策划、数值策划、战斗策划和角色设计,帮我组装一下。
AI:识别到已提供模块:系统/数值/战斗/角色。缺失模块:关卡/剧情/美术/QA/技术。
    开始构建模块索引表,缺失模块标注"待补产"。
    [执行六维度校验...]
    发现 3 个冲突,详见一致性冲突报告...
    术语表已汇总,共 24 个跨模块术语,其中 2 个需统一命名...
```

### 与单模块技能的协作
- 单模块策划产出时,可提前参考本技能的"校验维度",在产出时即避免常见冲突
- 组装时发现某模块内容不足,可返回对应单模块技能补产后再重新校验
- **各模块内容更新后,只需重新跑本技能的校验,不需要重建整个 GDD**(索引不变,只更新校验结果)

---

## 十二、与其他技能的协作

本技能产出的索引表、术语表和校验报告,通过引用语传递给团队和各模块:

- **模块索引表** → 查阅式引用 → 团队通过索引跳转查阅各模块内容
- **术语表** → 查阅式引用 → 各模块在产出时参考统一术语
- **校验报告** → 查阅式引用 → 各模块在修改时参考已知冲突和风险项
- **冲突报告** → 协作式引用 → 各模块根据冲突报告中的解决建议修改自身内容

> **引用方式说明:** 索引表和术语表为"查阅式引用"(本技能已产出,团队直接查阅);冲突报告中的解决建议为"协作式引用"(需要各模块回去修改自身内容)。

**前置依赖:**
- **建议前置:** `game-creative-discussion` 产出的"游戏创意框架"(作为项目总览摘要的来源)
- **必需前置:** 至少一个单模块策划案已产出(否则无内容可索引和校验)

**下游衔接:**
- 本技能产出的 GDD 框架文档作为团队协作的项目总览
- 校验报告作为各模块修改的指导
- 术语表作为各模块产出的命名参考
- **核心价值:** 事后跨模块校验,兜底解决各模块分散维护导致的不一致问题
