# 游戏创意框架完善

## 概述

本技能是**游戏创意框架完善工具**。无论用户是只有一句话创意,还是已有部分明确想法,**建议全流程都走本技能**——目的是快速完善一份用户确认的**"游戏创意框架"**,作为后续各策划模块细化的基础。

**核心价值:** 事前框架化,减少后期调整成本。如果跳过本技能直接进入各模块产出,各模块会各自假设玩家画像、动机、互联关系等,后期在 `game-full-workflow` 组装时大概率会因前期未框架化而导致大量返工。

**三层递进结构(基于交互设计五要素):**

```
第一层:策略分析(用户 / 目的 / 媒介)
  ├── 用户:玩家分析(Bartle 分群/认知模型/心理效应)
  ├── 目的:玩家动机(内部/外部动机/引导方向)
  └── 媒介:游戏设备环境预判(平台/输入/使用情境)
      → 产出:玩家画像 + 核心动机 + 媒介约束
        │
        ▼
第二层:创意发散(原有5维度,接收第一层输入)
  ├── 游戏类型与核心机制(← 媒介:玩法载体)
  ├── 目标受众与平台(← 用户+媒介:已部分确认)
  ├── 视觉风格与情绪体验(← 媒介:表现形式)
  ├── 核心循环与玩法深度(← 行为:玩家行为循环)
  └── 风险与可行性
      → 产出:初步设计大纲
        │
        ▼
第三层:功能规划(行为 / 场景细化 + 互联方向 + 自检)
  ├── 行为:核心玩法流程 + 操作反馈 + 模块互联方向表
  ├── 场景:特定场景下的游戏需求(设计思路三角 + 社交行为矩阵)
  └── 交互五要素自检
      → 产出:游戏创意框架(含策略要点 + 互联方向 + 自检报告)
```

**两种使用深度:**

| 用户情况 | 走法 | 深度 |
|---------|------|------|
| 想法模糊(仅一句话/关键词) | 从第一层完整走 | 发散讨论(深度引导) |
| 想法明确(已有部分信息) | 按快速启动模式跳过已有部分 | 确认+补全+框架化(不重写已有想法) |

**定位说明:**
- 本技能**负责完善游戏创意框架**,不直接产出策划案,不路由到后续模块
- 产出的"游戏创意框架"是后续各策划模块(系统/关卡/战斗/剧情/角色/数值)的**细化基础**,最终游戏内容在各模块内产出
- 用户可在**任意阶段**调用本技能:项目启动时、某个模块设计前、甚至项目中期方向迷茫时重新对齐
- **建议全流程都走本技能**——无论想法是否明确,都应在本技能产出一份用户确认的框架后再进入各模块细化,避免后期调整成本过大
- 若用户希望组装已产出的各模块策划案,请使用 `game-full-workflow`(GDD 组装器)

## 核心理念

- **框架先行**:先完善一份用户确认的"游戏创意框架",再进入各模块细化,减少后期返工
- **已有内容不重写**:用户已提供的想法直接采用,本技能只做确认、补全、框架化,不重新讨论已有内容
- **策略先行**:先明确"为谁做、为什么做",再发散"怎么做"
- **讨论优先于产出**:想法模糊时充分发散,想法明确时快速确认
- **引导式而非索取式**:AI 主动提出实现方向和问题,引导用户明确游戏性
- **框架是细化的基础,不是最终方案**:产出的框架供用户确认后,后续在各模块技能内细化
- **用户提供优先**:参考内容、案例、素材等由用户提供;若用户未提供,AI 基于游戏创意生成合适的内容
- **互联方向而非具体设计**:本技能产出的是各模块的高层互联方向,具体实施在各模块技能内细化

## 第一层:策略分析

基于交互设计五要素的**用户、目的、媒介**三个要素,做前置策略分析。这一层的产出会作为第二层创意发散的输入。

### 1.1 用户:玩家分析

#### Bartle 玩家分群

根据 Bartle 玩家类型理论,识别目标玩家主要属于哪些类型:

| 玩家类型 | 核心诉求 | 偏好的游戏元素 | 适合的游戏类型 |
|---------|---------|--------------|--------------|
| **探索型** | 发现未知、探索世界 | 隐藏内容、开放世界、秘密区域 | 开放世界、解谜、冒险 |
| **成就型** | 完成挑战、积累成就 | 成就系统、排行榜、收集要素 | RPG、模拟经营、策略 |
| **社交型** | 与他人互动、建立关系 | 社交系统、公会、合作任务 | MMO、社交游戏、派对 |
| **杀手型** | 竞争对抗、支配他人 | PVP、排行榜、竞技场 | MOBA、FPS、格斗 |

**引导方式:** 让用户从 4 种类型中选择 1-2 种主要类型,并描述吸引这些玩家的核心元素。

#### 认知模型(三层)

分析玩家在游戏中的认知过程,影响设计决策:

| 认知层 | 内容 | 设计影响 |
|-------|------|---------|
| **感官层** | 视觉、听觉、触觉 | 影响美术风格、音效、手柄震动等表现层设计 |
| **认知层** | 注意、记忆、理解 | 影响信息架构、UI 复杂度、教学引导设计 |
| **行为层** | 操作、反馈、学习 | 影响操作方式、反馈机制、难度曲线设计 |

#### 心理效应(影响玩家留存与付费)

识别游戏可利用的核心心理效应:

| 心理效应 | 应用场景 | 设计影响 |
|---------|---------|---------|
| **心流理论** | 挑战与技能的动态平衡 | 影响难度曲线、数值平衡目标区间 |
| **损失厌恶** | 玩家害怕失去已有资源 | 影响惩罚机制设计、付费点设计 |
| **沉没成本** | 玩家因已投入而不愿离开 | 影响长线留存设计、成长系统深度 |
| **社会认同** | 玩家受群体行为影响 | 影响社交系统、排行榜、流行度展示 |

**引导示例:**
```
AI:基于您的创意"音乐控制植物生长",目标玩家偏向?
  A. 探索型为主(发现新植物、解锁新旋律)
  B. 成就型为主(收集植物、完成图鉴)
  C. 探索+成就型并重
  D. 其他(请描述)

对应的认知模型侧重:
- A方案 → 感官层(视觉/听觉体验)优先
- B方案 → 认知层(信息架构/收集系统)优先
- C方案 → 三层平衡

对应的心理效应:
- A方案 → 心流理论为主(探索中的沉浸感)
- B方案 → 沉没成本为主(收集投入)
您的选择?
```

### 1.2 目的:玩家动机

分析玩家为什么玩这个游戏,影响核心循环和系统设计。

#### 决策动机(内/外部)

| 动机类型 | 具体动机 | 设计影响 |
|---------|---------|---------|
| **内部动机** | 好奇心 | 探索系统、隐藏内容、随机发现 |
| **内部动机** | 掌控感 | 自定义、建造、经营、策略决策 |
| **内部动机** | 成就感 | 挑战、成就、精通、收集完成 |
| **外部动机** | 奖励驱动 | 任务奖励、签到、成长资源 |
| **外部动机** | 竞争驱动 | PVP、排行榜、赛季奖励 |
| **外部动机** | 社交驱动 | 组队、公会、合作、分享 |

#### 引导设计方向

根据玩家动机,预判引导设计的重点:

| 引导类型 | 适用场景 | 预判重点 |
|---------|---------|---------|
| **新手引导** | 所有游戏必备 | 核心玩法教学、首单体验目标 |
| **功能引导** | 系统复杂时 | 新系统解锁时逐步介绍 |
| **付费引导** | F2P 游戏时 | 首充奖励、付费点设计时机 |

**引导示例:**
```
AI:基于探索型玩家定位,核心动机偏向?
  A. 内部动机为主(好奇心驱动探索)
  B. 内外部动机并重(探索 + 收集奖励)
  C. 其他(请描述)

对应的引导设计方向:
- A方案 → 新手引导强调"探索乐趣",付费引导弱化
- B方案 → 新手引导 + 功能引导 + 轻度付费引导
您的选择?
```

### 1.3 媒介:游戏设备环境预判

预判游戏的目标设备环境,影响后续技术选型和系统设计差异。

#### 设备环境分析表

| 维度 | PC端 | 主机端 | 移动端 | VR端 | 云游戏 |
|------|------|-------|-------|------|-------|
| 设备类型 | 个人电脑 | PS/Xbox/Switch | 手机/平板 | VR头显 | 跨端流式传输 |
| 输入方式 | 键鼠 | 手柄 | 触屏 | 体感/手柄 | 视设备而定 |
| 显示特性 | 大屏沉浸 | 大屏沉浸 | 小屏便携 | 全景视野 | 视设备而定 |
| 网络环境 | 稳定有线 | 稳定无线 | 移动网络 | 局域网 | 低延迟要求 |
| 使用情境 | 长时深度 | 客厅沉浸 | 碎片化场景 | 专属空间 | 随时随地 |
| 单局时长 | 30分钟+ | 30分钟+ | 5-15分钟 | 10-30分钟 | 视设备而定 |

#### 媒介约束对设计的影响

| 媒介约束 | 设计影响 | 传递给哪个下游模块 |
|---------|---------|-----------------|
| 移动端触屏 | 操作简化、UI 大按钮、手势操作 | `game-tech-implementation` 技术选型 |
| 碎片化场景 | 单局短、快速存档、随时可暂停 | `game-level-design` 关卡节奏 |
| 客厅沉浸 | 长时深度、社交同屏、大屏优化 | `game-narrative-design` 剧情深度 |
| VR 体感 | 沉浸优先、晕动症控制、3D 交互 | `game-tech-implementation` 引擎选型 |

**引导示例:**
```
AI:目标平台偏好?
  A. 移动端(碎片化,5-15分钟单局,触屏操作)
  B. PC端(深度体验,30分钟+,键鼠操作)
  C. 主机端(客厅沉浸,手柄操作)
  D. 跨端(请说明组合)

对应的媒介约束:
- A方案 → 移动端碎片化场景,核心循环要短平快
- B方案 → 深度体验,系统可复杂、剧情可深入
您的选择?
```

## 第二层:创意发散(原有5维度)

接收第一层策略分析的输入,按 5 个维度逐一讨论。

### 2.1 游戏类型与核心机制(← 媒介:玩法载体)

**讨论重点:** 基于**目的(玩家动机)**确定游戏类型,基于**媒介约束**适配核心机制。

**引导示例:**
```
AI:基于策略分析(探索型玩家 + 内部动机 + 移动端碎片化),
推荐游戏类型方向:
  A. 休闲解谜(匹配碎片化,探索新关卡)
  B. 轻度冒险(短单局,探索小世界)
  C. 收集养成(碎片化,收集驱动)
  D. 其他(请描述)

实现可行性:
- A方案:低(关卡化设计,机制可控)
- B方案:中(需小世界生成 + 短叙事)
- C方案:中(需大量素材 + 成长系统)
您倾向哪个方向?
```

### 2.2 目标受众与平台(← 用户+媒介:已部分确认)

**讨论重点:** 第一层已确认玩家画像和设备环境,此维度做补充和细化。

**引导示例:**
```
AI:第一层已确认探索型玩家 + 移动端平台。
补充讨论:
- 受众年龄区间?[全年龄/青少年/成人]
- 受众游戏经验?[休闲玩家/中度玩家/硬核玩家]
- 是否需要社交功能?[单机/异步社交/实时社交]
```

### 2.3 视觉风格与情绪体验(← 媒介:表现形式)

**讨论重点:** 视觉表现如何支撑情绪体验,结合**用户认知模型**(感官层优先级)。

**引导示例:**
```
AI:基于探索型玩家 + 感官层优先,推荐视觉风格:
  A. 水彩手绘风(温暖、柔和,治愈情绪)
  B. 低多边形(清新、简约,放松)
  C. 像素风(复古、可爱,怀旧)
  D. 其他(请描述或提供参考图)

情绪体验目标:
- 您希望玩家主要感受到:[放松/成就感/探索欲/创造欲]?
```

### 2.4 核心循环与玩法深度(← 行为:玩家行为循环)

**讨论重点:** 玩家在游戏中重复做什么,如何保持长期兴趣,结合**目的(玩家动机)**和**心理效应**。

**引导示例:**
```
AI:基于好奇心动机 + 心流理论,核心循环可能是:
  探索区域 → 发现新植物/旋律 → 解锁内容 → 扩展探索范围 → ...

玩法深度讨论:
- 短期目标(单局):[玩家在5-15分钟内要达成什么?]
- 中期目标(章节):[玩家在数小时内要解锁什么?]
- 铁长期目标(全游戏):[玩家通关后的追求是什么?]
- 心流区间:[挑战与技能的平衡点在哪里?]
```

### 2.5 风险与可行性

**讨论重点:** 实现风险、设计风险、市场风险,结合**心理效应**验证。

**引导示例:**
```
AI:基于前面的讨论,识别出以下潜在风险:
- 技术风险:[具体技术实现难点]
- 设计风险:[玩家体验问题,结合心理效应分析]
- 内容风险:[内容生产压力]
- 市场风险:[赛道竞争]

规避建议:
- [对应每个风险的规避方案]

您对哪些风险最关注?是否有补充?
```

## 第三层:功能规划

基于交互设计五要素的**行为、场景**两个要素,做功能层面的规划。这一层产出**模块互联方向表**,作为各下游模块细化的指导方向。

### 3.1 行为:玩家交互行为规划

#### 核心玩法流程

明确玩家的核心行为循环和操作方式:

| 行为要素 | 确认内容 | 示例 |
|---------|---------|------|
| 核心行为 | 玩家在游戏中主要做什么 | 探索 + 收集 + 解谜 |
| 操作方式 | 怎么操作(受媒介约束) | 触屏点击 + 滑动 |
| 反馈设计 | 操作后得到什么反馈 | 视觉特效 + 音效 + 数值变化 |
| 循环节奏 | 核心行为如何循环 | 探索 → 发现 → 解锁 → 扩展探索 |

#### 模块互联方向表(核心产出)

**这是本技能的核心产出之一。** 用一张表记录各模块的高层互联关系,作为各模块细化的指导方向。**只记录方向,不涉及具体设计。**

**模板:**

```markdown
## 模块互联方向表

### 核心玩法描述
[一段话描述玩家主要在做什么]

### 模块选取
- 核心模块:系统、数值、美术、QA、技术
- 类型模块(按需):[关卡/战斗/剧情/角色]

### 互联方向(各模块如何衔接)

| 模块A | ←→ | 模块B | 互联关系 | 细化指引 |
|-------|----|----|---------|---------|
| 关卡 | ←→ | 战斗 | [如:每关是一场对战] | game-level-design 把对战作为关卡核心;game-combat-design 定义对战机制 |
| 关卡 | ←→ | 剧情 | [如:每关卡推进一段剧情] | game-narrative-design 按关卡节点设计剧情 |
| 系统 | ←→ | 数值 | [如:成长数值随关卡递增] | game-numerical-design 平衡成长曲线 |
| 角色 | ←→ | 战斗 | [如:角色技能影响对战平衡] | game-character-design 考虑平衡性 |
| 系统 | ←→ | 关卡 | [如:解锁新内容作为关卡奖励] | game-system-design 设计解锁机制 |
```

**案例:对战模式作为关卡**

```markdown
## 模块互联方向表(示例)

### 核心玩法描述
PVP 对战为核心,玩家通过匹配进行对战,胜利解锁新内容。

### 互联方向

| 模块A | ←→ | 模块B | 互联关系 | 细化指引 |
|-------|----|----|---------|---------|
| 关卡 | ←→ | 战斗 | 每关是一场对战 | game-level-design 把"对战"作为关卡核心,细化匹配节奏、地图设计;game-combat-design 定义对战机制、技能、伤害公式 |
| 关卡 | ←→ | 数值 | 关卡难度=对手强度 | game-numerical-design 平衡对战数值、对手强度曲线 |
| 系统 | ←→ | 关卡 | 解锁新对战模式作为关卡奖励 | game-system-design 设计解锁机制 |
| 角色 | ←→ | 战斗 | 角色技能直接影响对战平衡 | game-character-design 角色设计考虑对战平衡 |
```

**平衡检查:** 三要素是否有一方明显过重或过轻?是否与玩家画像(第一层)匹配?

### 3.3 交互五要素自检

完成功能规划后,对照交互设计五要素做最终自检:

| 要素 | 自检问题 | 通过标准 |
|------|---------|---------|

## 快速启动模式

**核心原则:建议全流程都走本技能,深度按用户已有信息灵活调整。** 无论用户已有多少信息,都应在本技能产出一份用户确认的"游戏创意框架"后再进入各模块细化,以减少后期返工成本。

### 用户已有内容处理原则

**已有内容不重写、不重复讨论,只做确认和补全。** 具体处理方式:

| 用户已有内容 | 处理方式 |
|------------|---------|
| 玩家画像已明确 | 直接采用,不重新发散 |
| 游戏类型/核心机制已定 | 直接采用,仅确认与玩家画像/媒介是否匹配 |
| 视觉风格已定 | 直接采用,仅确认情绪目标 |
| 核心循环已定 | 直接采用,仅确认玩法深度和模块互联 |
| 任何缺失的维度 | 补讨论该维度 |

### 快速启动判断表

| 用户已提供 | 跳过的层 | 处理方式 |
|---------|---------|---------|
| 玩家画像 + 动机 + 平台 | 第一层全部 | 直接采用,进入第二层 |
| 玩家画像 + 平台 | 第一层用户+媒介 | 只补讨论"目的"(动机) |
| 游戏类型 + 核心机制 | 第二层对应维度 | 跳过"游戏类型与核心机制"维度,补其他维度 |
| 完整玩法(类型+机制+视觉+循环) | 第二层全部 | 直接采用,进入第三层做模块互联方向表 |
| 完整策略 + 玩法 | 第一层 + 第二层 | 直接进入第三层(行为规划+互联方向+自检) |
| 完整大纲(策略+玩法+互联) | 全部 | **不走本技能**,直接进入各模块技能产出策划案 |
| 仅创意描述 | 无 | 从第一层开始完整走 |

> **重要说明:** 即使是"完整大纲"场景,本技能的价值也在于**框架化确认**——把用户已有的点状信息整理成结构化的"游戏创意框架",并做交互五要素自检,确保各维度的完整性和模块互联的清晰度。**建议全流程都走本技能,区别只在于走的深度不同。**

### 快速启动示例

**示例 1:用户已有策略信息**
用户说:"我想做一个面向 Z 世代的移动端社交游戏,核心是合作探索,玩家可以自定义角色。"

→ AI 识别到已提供:
- 用户:Z 世代 + 社交型玩家
- 媒介:移动端
- 目的:合作(社交驱动)
- 场景:碎片化社交合作需求
- 行为:合作探索 + 角色自定义

→ 跳过第一层,直接进入第二层补全:
- 视觉风格(未提及)
- 核心循环细节(未提及)
- 风险与可行性(未提及)

→ 进入第三层:
- 模块互联方向表(角色←→社交系统←→关卡)
- 设计思路三角(情感优先)
- 社交矩阵(合作为主,表达为辅,竞争为可选)
- 五要素自检

**示例 2:用户已有完整玩法**
用户说:"我要做一个赛博朋克风格的动作 RPG,PC 端,核心机制是黑客入侵+近战格斗双战斗系统,核心循环是接任务→入侵/格斗→获取资源→升级黑客和格斗技能树→接更难的任务。"

→ AI 识别到已提供:
- 游戏类型:动作 RPG
- 核心机制:黑客入侵 + 近战格斗
- 视觉风格:赛博朋克
- 核心循环:接任务→战斗→升级→接更难任务
- 平台:PC 端

→ 跳过第二层全部,但第一层策略信息部分缺失:
- 补充讨论第一层:玩家画像(硬核动作玩家?Bartle 杀手型+成就型?)
- 补充讨论第一层:核心动机(成就感?竞争驱动?)

→ 直接进入第三层:
- 模块互联方向表(战斗←→数值←→关卡←→角色)
- 设计思路三角(操作优先,认知次之,情感辅助)
- 社交矩阵(如无社交,跳过)
- 五要素自检

**示例 3:用户已有完整框架**
用户说:"我们已经做好了完整的设计框架,包含玩家画像、核心机制、视觉风格、模块互联关系,现在想直接产出系统策划案。"

→ AI 识别到已有完整框架
→ **快速确认 + 五要素自检**(仍走本技能,但只做确认不做发散):
- 确认玩家画像、动机、媒介是否完整
- 确认模块互联方向表是否清晰
- 做交互五要素自检
- 若自检通过 → 产出"游戏创意框架"供用户最终确认
- 用户确认后 → 进入各模块技能产出策划案

> 即使是完整框架,也建议走本技能做一次确认和自检,避免后期各模块产出时才发现框架层面的问题。

## 整体流程

```
用户创意点(一句话/一段话/关键词/模糊方向)
    │
    ▼
第一层:策略分析(用户/目的/媒介)
    ├── 玩家分析(Bartle/认知模型/心理效应)
    ├── 玩家动机(内/外部动机/引导方向)
    └── 媒介预判(设备环境/使用情境)
    │
    ├── 信息不足 → 继续引导讨论
    └── 信息充分 → 进入第二层
    │
    ▼
第二层:创意发散(5维度,接收第一层输入)
    ├── 游戏类型与核心机制(← 媒介)
    ├── 目标受众与平台(← 用户+媒介)
    ├── 视觉风格与情绪体验(← 媒介)
    ├── 核心循环与玩法深度(← 行为)
    └── 风险与可行性
    │
    ├── 讨论不充分 → 继续引导
    └── 讨论充分 → 进入第三层
    │
    ▼
第三层:功能规划(行为/场景细化 + 互联方向 + 自检)
    ├── 行为规划(核心玩法流程/操作反馈/模块互联方向表)
    ├── 场景确认(设计思路三角/社交行为矩阵)
    └── 交互五要素自检
    │
    ├── 自检未通过 → 回到对应层补充
    └── 自检通过 → 整理"游戏创意框架"
    │
    ▼
用户确认框架
    ├── 不确认 → 标注修改点 → 回到对应层
    └── 确认 → 流程结束,框架交付
                    │
                    └─→ 用户基于框架进入各模块技能细化
                         → 最终由 game-full-workflow 组装为完整 GDD
```

## 与其他技能的协作

本技能产出的策略分析要点和模块互联方向表,会通过引用语传递给下游模块:

- **玩家画像(含 Bartle 分群)** → 查阅式引用 → `game-character-design` 角色定位参考
- **核心动机/心理效应(心流区间)** → 查阅式引用 → `game-numerical-design` 数值平衡目标
- **设备适配要求** → 查阅式引用 → `game-tech-implementation` 技术选型参考
- **社交矩阵定位** → 查阅式引用 → `game-system-design` 社交系统设计参考
- **设计思路三角平衡** → 查阅式引用 → 各策划模块的体验平衡参考
- **模块互联方向表** → 查阅式引用 → 各策划模块的细化指导方向

> **引用方式说明:** 以上均为"查阅式引用"(本技能已产出内容,下游模块直接查阅),不是"协作式引用"(需要下游产出)。下游模块在产出时查阅本技能的大纲即可获取方向指导。

**前置依赖(无):** 本技能是最前置环节,无前置依赖。用户可直接从本技能进入。

**下游衔接:**
- **建议流程:** `game-creative-discussion` 产出用户确认的框架 → 各单模块技能基于框架细化产出策划案 → `game-tech-implementation` 对接技术规格 → `game-full-workflow` 组装并校验一致性 → 完整 GDD
- **已有各模块内容:** 用户直接带着已有各模块内容进入 `game-full-workflow` 组装(此时可省略本技能,但建议先用本技能做一次框架确认和五要素自检)
- **核心价值:** 事前在本技能完善框架,减少事后在 `game-full-workflow` 校验时发现的冲突数量
